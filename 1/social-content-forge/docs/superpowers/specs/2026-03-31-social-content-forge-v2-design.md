> 日期：2026-03-31

## 一、背景与目标

**项目定位**：将任意输入转化为多平台高质量爆款内容

**v2 核心升级目标**：

1. 热点发现 — 实时获取多渠道热点，赋能选题
2. 自我进化生成 — 让内容生成具备「学习-评审-改进」能力
3. 质量门控 — 不达标不输出，确保最终内容质量
4. 素材增强 — 解决「主题搜索」效果差的问题

**目标内容风格**：Dan Koe 风格，姜胡说为质量参照

- 受众：大厂边缘人、小企业主、职场内卷挣扎者
- 特点：个人故事切入、真实洞察、反常识观点、理论+实操、长文深度

## 三、热点发现层

### 3.1 一期数据源

| 数据源 | 热点内容 | 接入方式 | 更新频率 |
|--------|---------|---------|---------|
| 微博热榜 | 实时热搜榜 | 第三方API/爬虫 | 实时 |
| Twitter Trending | Trending Topics | 官方API/第三方 | 实时 |
| Google Trends | 搜索趋势/热门查询 | 官方API | 准实时 |
| 小红书热榜 | 热榜话题/搜索热词 | 第三方数据/爬虫 | 准实时 |
| Reddit | Popular/Growing | 开放可爬 | 准实时 |

### 3.2 热点数据结构

```typescript
interface HotTopic {
  id: string;
  platform: 'weibo' | 'twitter' | 'google' | 'xiaohongshu' | 'reddit';
  title: string;           // 热点标题
 热度值?: number;          // 平台热度指标
  category?: string;       // 分类：科技/职场/娱乐...
  link?: string;           // 原始链接
  fetchedAt: Date;
}
```

### 3.3 热点融合策略

- 多平台热点去重合并
- 按分类聚合，优先推荐与目标受众相关的热点（职场、财富、认知提升）
- 支持按平台筛选和全局热度排序

## 四、内容理解层

### 4.1 子模块职责

| 子模块 | 输入 | 输出 | 目的 |
|--------|------|------|------|
| **素材解析** | 原始素材（URL/文本/想法） | 结构化素材包 | 把混乱信息变成可用材料 |
| **爆款解构** | 同类爆款案例 | 爆款公式/叙事结构 | 提炼「这个主题的爆款模板」 |
| **情绪地图** | 主题 | 情绪触发点清单 | 找到能引发强烈情绪的具体点 |
| **竞品分析** | 同主题已发布内容 | 差异化切入点 | 找到还没被说透的角度 |
| **受众画像** | 主题 | 核心受众+边缘受众 | 知道写给谁看、谁会转发 |
| **生命周期判断** | 主题+时效性 | 内容生命周期预测 | 判断是速爆还是长尾 |

### 4.2 素材增强模块（解决主题搜索效果差）

**问题根因**：用户输入「一句话/短想法」，缺乏具体素材，模型只能泛泛而谈。

**解决方案**：

```
用户输入（一句话想法）
      ↓
┌─────────────────────────────────────┐
│  素材增强模块（内部自动执行）          │
│  1. 搜索同主题爆款内容               │
│  2. 提取其中的金句/案例/数据         │
│  3. 反向注入到生成Prompt             │
└─────────────────────────────────────┘
      ↓
融合后的丰富Prompt → 生成 → 评审 → 输出
```

**效果预期**：

- 用户输入还是简单的一句话
- 模型拿到「一句话 + 爆款参考素材包」
- 生成内容从「泛泛而谈」变成「有具体支撑的洞察」

## 五、两层 Prompt 体系

### 5.1 架构

| 层 | 说明 | 优化方式 |
|----|------|---------|
| **基础Prompt** | 六/九维度定义、评审标准、格式规范、Dan Koe风格约束 | GitHub调研+姜胡说案例分析+人工维护 |
| **动态Prompt** | 素材包+改进建议+任务背景+受众信息 | 每轮生成后自动演进 |

### 5.2 基础Prompt核心要素

基于 Dan Koe风格 + 姜胡说内容特征：

- **开头**：个人故事/脆弱性切入，不用公式化开头
- **视角**：第一人称「我」的实战经历
- **洞察**：反常识观点，挑战主流观念
- **案例**：具体人物/公司/数据（冯小刚、贝索斯、张一鸣等）
- **结构**：递进式章节（诊断→原因→解决）
- **语言**：口语化、有节奏、短句强调

### 5.3 动态Prompt注入

每轮生成后，动态Prompt更新：

```typescript
interface DynamicPrompt {
  taskBackground: string;      // 本次任务背景
  materialPackage: Material[];  // 素材包（来自素材增强或爆款解构）
  improvementSuggestions: string[]; // 上轮评审的改进建议
  targetAudience: Audience;    // 受众画像
}
```

## 六、自我进化生成层

### 6.1 流程

```
生成内容
    ↓
内嵌评审（在同一LLM调用中，不单独打分，节省token）
    ↓
任一维度<5？ → 是 → 触发改进
    ↓
加权总分≥85？ → 是 → 输出
    ↓ 否
改进建议注入动态Prompt → 重试（同一LLM，最多1次）
    ↓
还不达标 → 换下一个LLM → 新prompt带入改进思路
    ↓
所有LLM失败 → "生成达不到要求，已停止任务"
```

### 6.2 LLM轮询策略

```typescript
interface LLMConfig {
  name: string;           // 模型名称
  strength: string[];     // 擅长的内容类型
  maxRetries: number;    // 最大重试次数（默认1）
}

// 路由优先级：按擅长类型+评分历史动态选择
const llmPool: LLMConfig[] = [
  { name: 'claude', strength: ['深度长文', '个人洞察'], maxRetries: 1 },
  { name: 'gpt', strength: ['结构清晰', '实操指南'], maxRetries: 1 },
  { name: 'deepseek', strength: ['快速生成', '观点鲜明'], maxRetries: 1 },
];
```

### 6.3 内嵌评审设计

不单独调用评分API，而是在生成prompt末尾附加评审指令：

```
【评审要求】
1. 检查是否有人个人故事/经历切入
2. 检查是否有反常识洞察
3. 检查是否有具体案例/数据支撑
4. 检查单维度是否有<5分项
5. 如不达标，给出具体改进建议
```

## 七、九维度质量评估层

### 7.1 维度定义

| 维度 | 权重 | 评估内容 | 否决条件 |
|------|------|---------|---------|
| **情绪激发度** | 20% | 能否引发强烈情绪反应 | <5一票否决 |
| **实用价值** | 20% | 读者能得到什么具体好处 | <5一票否决 |
| **叙事结构** | 15% | 故事是否引人入胜，开头是否有钩子 | - |
| **社交货币** | 10% | 转发能彰显转发者什么身份 | - |
| **争议引导** | 10% | 能否引发讨论而非沉默 | - |
| **时效贴切** | 5% | 是否契合当前热点/趋势 | - |
| **差异化程度** | 10% | 和同类内容有什么不同 | - |
| **可转发场景** | 5% | 读者在什么场景会转发 | - |
| **转化潜力** | 5% | 能否推动关注/互动/行动 | - |

### 7.2 质量门控

```
加权总分 ≥85 → 通过，输出
任一维度 <5 → 一票否决，不输出，触发改进
```

## 八、平台适配层

### 8.1 各平台内容要求

| 平台 | 字数 | 结构 | 特殊要求 |
|------|------|------|---------|
| 微信公众号 | 2000-3000字 | 开头钩子→方法论→案例→金句→CTA | 个人故事切入，深度长文 |
| 小红书 | 600-1000字 | 封面文字+标签+情绪化正文 | 强情绪、个人体验感、种草感 |
| Twitter | ≤280字 | 观点鲜明、前3字抓人 | 可扩展为Thread |

### 8.2 平台适配原则

- 不是「同一内容改改格式」，而是「同一洞察用不同表达方式」
- 微信侧重深度，Twitter侧重观点，小红书侧重情绪

## 九、输出层

### 9.1 输出格式

```typescript
interface ContentOutput {
  id: string;
  source: string;              // 原始输入
  topics: string[];            // 关联热点
  generatedAt: Date;
  qualityScore: number;        // 最终评分
  content: {
    wechat: PlatformContent;   // 微信公众号完整内容
    xiaohongshu: PlatformContent; // 小红书完整内容
    twitter: PlatformContent;   // Twitter完整内容
  };
  metadata: {
    llmUsed: string;           // 最终使用的LLM
    iterations: number;        // 生成迭代次数
    materialSources: string[]; // 素材来源
  };
}

interface PlatformContent {
  title: string;
  body: string;                // 完整Markdown
  tags?: string[];            // 标签（小红书）
  coverText?: string;          // 封面文字（小红书）
}
```

### 9.2 飞书同步

同步完整三平台 Markdown 内容到多维表格，而非零散字段。

### 9.3 SQLite存储

本地持久化，支持历史记录查询和质量分析。

## 十一、风格学习库

### 11.1 设计理念

「日积跬步，日拱一卒」—— 通过持续学习爆款/差案例，不断校准内容质量标准，让系统的内容质量随时间不断提升。

### 11.2 库结构

```
docs/style-library/
├── library.json              # 案例索引和元数据
├── good/                     # 爆款案例库
│   ├── 2024-12-09-姜胡说-赚钱最重要的是什么.md
│   └── ...
├── bad/                      # 差案例库
│   └── 2024-xx-xx-视频播放量上不去.md
└── insights/                 # 学习洞察（每次学习后的总结）
    └── 2026-03-31-insights.md
```

### 11.3 案例文件格式

```markdown
---
type: good|bad
source: 姜胡说公众号
url: https://mp.weixin.qq.com/s/...
addedAt: 2026-03-31
tags: [赚钱, 认知, 方法论]
---

# 正文内容...
```

### 11.4 学习机制

```
【每次项目启动时自动执行】

1. 检查 docs/style-library/ 是否存在新增案例
   ↓
2. 如有新案例，自动学习并总结：
   - 解构爆款/差在哪
   - 提炼可复用的写作手法
   - 更新内容质量标准
   ↓
3. 输出学习总结报告，找用户确认
   ↓
4. 用户确认后，更新基础Prompt/内容质量标准
```

### 11.5 工作流程

```
用户添加新案例到 style-library/good/ 或 style-library/bad/
        ↓
下次启动项目（cd social-content-forge）
        ↓
自动检测到新案例
        ↓
读取并学习案例内容
        ↓
生成学习洞察报告：
  - 「这个爆款好在...」
  - 「提炼出的写作手法：...」
  - 「建议更新到质量标准的...」
        ↓
找用户确认：「学到了这些，要不要更新质量标准？」
        ↓
用户确认 → 更新 docs/memory/content-quality.md
用户否定 → 记录反馈，但不更新
```

### 11.6 案例与洞察分离原则

```
docs/style-library/
├── insights/                 # ★ 核心资产，只增不减
│   └── 2026-03-31-洞察.md   # "爆款好在...差在哪..."
│
├── good/                     # 爆款原文（可删可增）
│   └── ...
│
└── bad/                      # 差案例原文（可删可增）
    └── ...
```

**管理原则**：

| 类型 | 可删减？ | 说明 |
|------|---------|------|
| **insights/** | 否 | 核心学习成果，只增不减 |
| **good/** | 可 | 案例原文，删减不影响洞察 |
| **bad/** | 可 | 案例原文，删减不影响洞察 |

**定期合并流程**：
1. 每月将 `insights/` 的结论合并到基础 Prompt
2. 合并后清空 `insights/`，重新积累
3. `good/` 和 `bad/` 中的案例可保留备查，也可清理

### 11.6 CLAUDE.md 启动指令

在 `social-content-forge/CLAUDE.md` 中配置：

```markdown
## 启动时检查

每次 cd 到本项目目录时：
1. 检查 docs/style-library/ 是否有新增案例
2. 如有，输出：「检测到 N 个新案例，正在学习...」
3. 学习后输出总结，找用户确认
4. 确认后更新内容质量标准
```
