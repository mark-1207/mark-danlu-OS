# Social Content Forge — 相似度验证模块设计

**日期**: 2026-04-01
**状态**: 已确认
**模块**: social-content-forge / evaluator

---

## 背景

当前 social-content-forge 的流程为：

```
输入 → 提取 → 理解 → 评估(六维度) → 适配(微信/小红书/Twitter) → 同步飞书
```

在适配生成后、飞书同步前，需要插入「相似度验证」环节，解决以下问题：

- **版权/原创性**：避免照搬原文案例被读者识破
- **账号人设一致性**：改写后的语调与人设有差异
- **平台去重需求**：各平台算法会识别高度相似内容

---

## 验证维度与阈值

| 维度 | 阈值 | 说明 |
|------|------|------|
| 案例相似度 | ≤20% | 实体（人名/公司/产品/具体事件）重叠率 |
| 金句照搬 | ≤15% | 原文原话连续超过15字视为照搬 |
| 语义相似度 | ≤70% | 改写内容与原文的 Embedding 余弦相似度 |
| 标题差异度 | ≥60% | 改写后标题与原文标题的编辑距离/差异程度 |
| 开头结尾原创性 | ≥50% | 开头结尾段落与原文的语义差异程度 |

**阈值级别**：适中（>50%触发为严格，>70%触发为宽松）

---

## 判定规则

### 双轨制

| 触发条件 | 规则 |
|---------|------|
| 总分超标 | 加权相似度 >70%（案例30% + 金句20% + 语义30% + 标题10% + 开头结尾10%） |
| 单维超标 | 任一维度超过阈值 |
| **两者满足其一** | → 不通过，触发优化 |

---

## 流程

```
适配生成
    ↓
质量评分 ≥80？ → 否 → 先优化质量
    ↓ 是
相似度验证（一次性检测5个维度）
    ↓
┌─ 不通过 → 自动重新优化该维度（无用户通知）
│        → 再次验证
│        ↓
│      ...持续直到通过...
│
└─ 通过 → 通知用户 + 展示改动摘要 → 同步飞书
```

### 优化轮次
- **不设上限**，持续优化直到通过
- 每次优化后重新验证

---

## 用户交互

### 优化过程中
- **不打扰用户**，静默完成迭代

### 每次通过后
- **主动展示改动摘要**，格式如下：
  > "案例已替换为你的[XX故事]，金句已改写，语义相似度从75%降至42%，通过验证，已同步飞书"

### 追问机制
- 用户可主动追问："这次改了什么？" → 展示详细报告
- 某维度长期不通过 → 记录并可追问

---

## 实施注意事项

1. **语义相似度**依赖 Embedding API（OpenAI/Cohere/国内模型），需配置 API Key
2. **案例实体提取**使用 NER（命名实体识别），可调用 LLM 或本地模型
3. 验证不通过时，优化指令应精准指向超标维度，避免全局重写
4. 飞书同步使用已配置的 `FEISHU_APP_TOKEN` 和 `FEISHU_TABLE_ID`
