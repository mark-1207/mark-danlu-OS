场景一
skill "帮我写..." → 原创生成 → Obsidian 回流
用户输入自然语言需求，系统全自动生成多平台适配内容，写入 Obsidian

触发方式：CLI `skill "帮我写一篇关于XXX的文章 发公众号"` 或 Claude Code 里直接说需求

用户痛点：有写作需求但不知道从哪开始，需要 AI 帮助构思标题、大纲和内容

场景方案：skill 解析意图后自动走 6 步 pipeline，最终生成公众号/小红书/抖音三平台内容

Step 1 — Intent 解析

1 parseIntent() — 判断 intent 类型
从用户输入提取关键词 + 目标平台。
有 .md 文件路径 OR 有意图词（二创/改写/爆款/rewrite）→ recreate
否则 → create。
自动识别平台词（公众号/小红书/抖音）。
输入: "帮我写一篇关于AI时代的职场人文章 发公众号" intent.type: "create" intent.keyword: "AI时代的职场人" intent.platforms: ["wechat"]
→ 进入 create pipeline

Step 2 — Topic Analysis

2 topic-analysis — 主题深挖
LLM 分析选题方向，提取关键词 tags，生成爆款结构建议。
输出供后续步骤使用的主题分析结果。
topic-analysis: { tags: ["AI时代", "职场人", "职业危机", ...], viralStructure: "对比型叙事结构", topicAngle: "从普通人视角看AI带来的职业安全感缺失" }

Step 3 — Outline 生成

3 outline-wechat — 大纲 + 知识迁移规划
从 Obsidian 40_知识库/ 加载相关素材（原子库/洞察库/金句库/思维模型等）。
大纲阶段规划每个 section 使用哪个素材、怎么用（knowledgeTransfer）。
遵循五步波纹法情绪曲线设计。
outline-wechat: { sections: [ { title: "那个被AI'优化'掉的下午", knowledgeTransfer: "案例：朋友老李的故事" }, { title: "AI冲击不是意外，是系统重构", knowledgeTransfer: "数据：麦肯锡报告" }, ... ], emotionalCurve: ["钩子→铺垫→冲突→共鸣→CTA"] }

Step 4 — 素材搜索

4 material-search — Web + Obsidian 素材
按大纲规划的每个知识点搜索 Web 素材。
同时注入 Obsidian 素材（知识迁移方案B：动态激活）。
合并去重后供 content 阶段使用。
material-search: { wechat: 1, // 找到1条素材 xiaohongshu: 0, douyin: 0 } Obsidian materials injected: 8 cards

Step 5 — 内容生成 + 审查

5 content-wechat → review-wechat
按大纲 + 素材生成正文，执行六维质量规范（温度/热度/深度/厚度/情绪曲线/知识迁移）。
review-wechat 对初稿进行质量审查，不通过则重写。
✓ 生成完成 tokens: +9326

Step 6 — Obsidian 回流

6 ObsidianWriter.writeArticle() — 文章入库
生成完成后自动写入 Obsidian 50_资源/生成文章/。
用户可手动拆解沉淀为原子卡，下次创作时调用。
写入路径: d:\软件\obsidian笔记\内容素材库\50_资源\生成文章\ AI不是电钻是筛子：30岁后如何构建反脆弱职业系统.md
✓ 已同步 Obsidian

场景二
skill "二创：d:/文章.md" → 差异化二创 → 原创性审查
用户提供已有文章路径，系统分析爆款要素、生成差异化方向、产出新内容并进行原创性审查

触发方式：`skill 二创：d:/文章.md` 或 `skill 帮我改写这篇文章` 或 `node dist/index.js recreate -i d:/文章.md`

用户痛点：有一篇好文章，想以不同角度/受众/风格改写，但没有思路

场景方案：系统先解剖原文爆款要素，再生成 3 个差异化方向，用户选择后生成新内容，最后审查原创性

Step 1 — 原文验证

1 validateAndCleanInput() — 输入验证 + 清理
读取 .md 文件，检测 HTML 内容并自动清理。
验证通过后提取正文供后续 pipeline 使用。
originalArticle: "（提取的正文内容）" validation: { wasHtml: false, errors: [], warnings: [] }

Step 2 — 病毒拆解

2 viral-deconstruction — 解剖原文爆款要素
分析原文的：钩子技术、情绪曲线、叙事结构、金句、案例。
输出 viral-genome（爆款基因组），供后续差异化方向和新大纲使用。
viral-genome: { hookTechnique: { type: "判断式金句钩子", mechanism: "..." }, emotionCurve: [...], narrativeStructure: [...], powerSentences: [...] }

Step 3 — 差异化方向

3 viral-differentiation — 生成 3 个差异化方向
三个维度生成差异化方向：视角转移、受众转移、内容转移。
interactive 模式下暂停等待用户选择；auto 模式自动选最优。
directions: [ { name: "受众降维", newAngle: "从职场人视角降到普通家庭主妇", perspectiveShift: "高管→普通人", compositeScore: 8.2 }, { name: "观点反转", newAngle: "不是危机，是机会", contentShift: "悲观→乐观", compositeScore: 7.5 }, ... ]
⚠ interactive 模式暂停，等待用户选择方向

Step 4 — 双重审查

4 dual-review — 原创性审查 + 爆款潜力评估
originalityReport：检测正文与原文相似段落（goldQuote/caseStudy/data 类型），输出相似度。
viralPotentialReport：评估新内容的六维质量评分。
dual-review: { originalityReport: { overallScore: 7/10, passThreshold: true, flaggedParagraphs: [ { type: "caseStudy", similarity: 0.82, paragraphIndex: 2 } ] }, viralPotentialReport: { scores: {...} } }

Step 5 — 元素级优化（条件触发）

5 local-rewrite — 针对高相似段落重写
当 flaggedParagraphs 有内容时，提示用户选择要替换的段落编号。
触发 rewrite-actions 原子级重写（rewrite-title/hook/section/cta 等）。
成本控制：超过 maxCostPerRun 时跳过。
输入要替换的编号（逗号分隔，如 1,3），回车确认，0 跳过: 2 已选中：[案例] 相似度 0.82 → rewrite-section: 正在重写...

Step 6 — post-gen 提示 + corpus 保存

6 askPostGen() — 生成后提示
二创完成后，TTY 环境下询问用户：接受 / 修改 / 放弃。
修订（revise） → 进入修订 pipeline（场景三）。
同时保存到 corpus/original/ 版本历史，供碎片库学习。
askPostGen() 选项: [accept] — 接受当前版本，退出 [revise] — 修订（进入 RevisionPipeline） [abort] — 放弃 corpus/original/ 保存格式: recreate_2026-05-19_AI不是电钻是筛子.md recreate_2026-05-19_AI不是电钻是筛子_v1.md ← 上一版

场景方案：精准定位要改的元素（标题/Hook/正文/CTA/案例/金句），定向重写，循环确认直到满意

场景四
topic scrape → 飞书 → learn --analyze → 碎片提取 → Obsidian
用户输入竞品文章 URL，系统抓取 → 写入飞书 → AI 分析 → 碎片提取 → Obsidian 原子库

触发方式：`topic scrape --url "URL"` 或 `topic scrape --file urls.txt`

用户痛点：需要持续跟踪竞品内容，但手动抓取费时费力，分析和碎片提取更是负担

场景方案：一键批量抓取 → 自动写入飞书 → learn 命令完成 AI 分析和碎片提取

Step 1 — 竞品抓取

1 topic scrape — 抓取 + 写入飞书
支持单条 `--url` 或批量 `--file urls.txt`。
抓取正文、去 HTML、写入飞书竞品表（来源类型=竞品抓取，状态=待分析）。
错误跳过继续，进度报告。
# 单条 node dist/index.js topic scrape --url "https://mp.weixin.qq.com/s/xxx" # 批量 node dist/index.js topic scrape --file urls.txt # 桌面一键（双击运行） # 竞品抓取.bat
✓ 写入飞书（状态: 待分析）

Step 2 — AI 分析

2 learn --analyze — AI 提取爆款结构/角度/标签
读取飞书待分析记录，调用 LLM 分析：
爆款结构、叙事结构（故事型/清单型/对比型/分析型/混合型）、情感调性（励志/冷静/温暖/犀利/幽默）、内容角度、标签。
分析完成后更新飞书记录（状态=已分析），同时补填叙事结构/情感调性/内容角度字段。
# 执行 node dist/index.js learn --analyze # 分析字段 { 爆款结构: "对比型开头+清单型展开", 叙事结构: "对比型", 情感调性: "犀利", 内容角度: "反常识颠覆", 标签: ["职场", "效率", "反常识"] }

Step 3 — 碎片提取

3 learn --extract-fragments — 拆解为原子卡写入 Obsidian
读取飞书已分析记录，按句子/段落拆解为原子卡。
每个原子卡：类型标签（atom）、质量评分、使用次数统计。
写入 Obsidian 40_知识库/原子库/，状态更新为"已入库"。
# 执行 node dist/index.js learn --extract-fragments # 原子卡写入路径 40_知识库/原子库/ ├── 01_观点/ # viewpoint 类型 ├── 02_案例/ # case 类型 ├── 03_金句/ # quote 类型 └── 04_原理/ # principle 类型 # 原子卡 frontmatter --- type: atom subtype: viewpoint status: active topics: [职场, 效率] quality_score: 8 usage_count: 0 created: 2026-05-19 ---
✓ 状态更新为"已入库"

Step 4 — 竞品分析报告

4 learn --include-competitor — 生成竞品 vs 我方对比报告
读取竞品表（analyzed/stored）和反馈表，计算各维度平均互动率。
输出 6-section 完整报告（含核心发现、市场格局、高表现规律、差距缺口、洞察、下一步行动）。
写入 output/corpus/竞品库/ + 同步 Obsidian 竞品库/。
# 执行 node dist/index.js learn --include-competitor # 输出 output/corpus/竞品库/竞品分析报告-2026-05-19.md → 同步到 Obsidian 竞品库/

ContentForge 完整命令 & 边界

命令作用边界 / 限制
skill "需求描述" 自然语言触发，解析为 create/recreate 有路径或意图词（二创/改写/爆款）即触发二创，无需同时满足
recreate -i <file> 爆款二创，支持 --interactive / --platforms interactive 模式需 TTY 环境
revise <runId> 修订已有 run，支持 --list / --revert 需要传入有效 parentRunId
topic scrape --url/--file 竞品抓取，写入飞书 URL 需要可公网访问
learn --analyze 批量 AI 分析飞书待分析记录 需要 lark-cli 认证
learn --extract-fragments 从已分析记录提取原子卡 碎片为 0 时写 fallback 卡
learn --include-competitor 竞品分析报告 需要竞品表有 analyzed 记录
learn --feedback-summary 我方内容分析（console） 需要反馈表有数据
learn --backfill-analysis 竞品历史记录回填 一次性操作
