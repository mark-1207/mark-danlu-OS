# 99 — 经验教训沉淀（Lessons Learned）

> 目的：把"被骂过 / 被纠正过 / 被验证过"的经验沉淀下来，不仅适用于炉，也可在其他项目复用。
>
> 维护规范：每次被纠正/被验证后立刻新增一条；格式 = 规则 + Why + How to apply + 来源/日期。
> 编号规则：F = 用户反馈；D = 开发习惯；P = 炉项目专属；E = 通用经验。

## 0. 思考方法（最重要的 1 条）

### 规则 0：不要直接给方案，先思考

任何"如何设计 X / 如何优化 Y"的问题，先回答 3 步：
1. **现状是什么**（这个环节真正在做什么、做到什么程度）
2. **问题在哪里**（不只列表现，挖根因、挖深层结构性问题）
3. **方向和框架**（用什么样的工具/模型/视角去思考，不直接给实现）

只有在 3 步都清晰后，才能给方案。方案要明确说"这是我思考后的判断，不是默认动作"。

### Why
用户反复被"上来就 ABCDE 方案"或"上来就 Round 1/2/3"激怒。原因是没真正理解清楚现状和问题，给的方案是结构性重组，不是深度分析。

### 反模式
- 上来就给"3 轮 7 问"这种结构
- 上来就列"A 方案 / B 方案 / C 方案"
- 上来就给"我会这样做：第一...第二..."
- 不挖根因，只列表现
- 用"我会..."开头而不是"现状是..."开头

### 正模式
- 先承认"现状是什么"
- 再挖"问题真正在哪里"
- 再谈"用什么思维工具去思考"
- 最后才是"基于以上分析，我的方案是..."

### 来源
- 日期：2026-06-15
- 反馈原文："你要思考 1. 现状 2. 提升方向 3. 怎么做才能价值最大化。这些是你要思考的，不是直接上来给我方案。请你改掉不思考就直接出方案的问题"
- 触发场景：苏格拉底追问环节设计

## 1. 用户反馈层（F1-F11）

### F1：审计必须用 Read/Grep 验证代码

**规则**：任何"缺失"判断必须先用 Grep/Read 验证代码不存在。

**Why**：只读文档不读代码就列"差异表"是常见错误——文档与代码脱节时结论全错。

**How to apply**：列"修复计划"前，必须先有"Grep 验证命令 + 结果"作为前置证据。

**来源**：CLAUDE.md 错误 2

---

### F2：完整流程必须用 run 入口

**规则**：所有 PRISM-OS 流程必须用 `python prism_os.py run "<命题>"` 触发，禁止手动拼凑单阶段。

**Why**：直接调 prism/gap/ccos/narrate 会跳过 Phase 0/1/1.5，破坏流程完整性。

**How to apply**：用户在流程中只能看到"run 一个命令"这一个入口；单阶段命令是开发者 debug 工具，不是用户路径。

**来源**：被骂过 5 次，记录在 `feedback_run_is_canonical_entry` / `feedback_dont_skip_flow_again` / `feedback_prismos_canonical_flow`

---

### F3：内容生成不要用 --auto/--no-interactive

**规则**：交互流程默认开启，CI/调试场景才用 `--no-interactive` 或 `--skip-gateway`。

**Why**：交互是用户体验的一部分，自动化会绕过用户的 7 类追问/标题选择/Gap 决策。

**How to apply**：CLI 默认 `interactive=true`；用 `echo "skip"` 模拟用户输入是反模式。

**来源**：`feedback_no_auto_for_creation`

---

### F4：每项 commit 必做端到端验证

**规则**：每项 commit 后跑全量 test + tsc/type check + Lint。

**Why**：半成品 commit 会让后续追溯困难，破坏"代码能跑 + 测试通过"的承诺。

**How to apply**：commit 前 3 步：
1. `npm test`（或 pytest）全过
2. `tsc --noEmit`（或 mypy）全过
3. 关键路径手动跑一遍

**来源**：`feedback_e2e_verify_before_next`

---

### F5：内容风格 = 自媒体深度解析

**规则**：犀利观点 + 接地气 + 情感共鸣。避免"赋能""在这个时代""让我们来看看""不可否认"等套话。

**Why**：官方口吻 + 数据堆砌 = AI 味儿重，用户最厌恶。

**How to apply**：
- 写"在这个时代..." → 改为"这两年..."
- 写"让我们来看看..." → 改为"举个例子..."
- 写"不可否认的是..." → 直接删掉

**来源**：`feedback_content_style` + 炉项目 L1 禁词扫描

---

### F6：严格执行方案计划 + TDD + 端到端验证

**规则**：任何实施前先对齐方案；写代码前先写测试；commit 前端到端验证。

**Why**：自以为是、想象、跳步骤会导致方案跑偏，最后交付时大量返工。

**How to apply**：用三段式检查清单：
- 方案：文档有吗？有最新吗？有 checklist 吗？
- 代码：测试在先吗？单测 + 端到端都写了吗？
- 验证：跑过吗？通过吗？复盘有吗？

**来源**：`feedback_strict_plan_tdd`

---

### F7：独立项目不要混在一起描述

**规则**：多项目融合时，明确"物理隔离 + 能力借鉴"的边界。

**Why**：PRISM-OS 和 ContentForge 是独立项目，混在一起描述会模糊边界，导致能力归属不清。

**How to apply**：
- 描述项目时，明确说"这是 X 项目 / Y 项目 / Z 项目"
- 能力归属要可追溯（哪个能力在哪个项目）
- 跨项目能力借鉴 = 手动复制 + 决策

**来源**：`feedback_prismos_contentforge_separate`

---

### F8：质量门控不能太严

**规则**：任何评分/质检环节，留出 1-2 个 fallback 路径。

**Why**：门控太严格导致失败过多，健壮性差。

**How to apply**：
- 评分失败 → 默认走"低分警告"而非"硬中断"
- 质检失败 → 给"修复建议"而非"直接拒绝"
- 留 1-2 个"放行条件"作为 robust parsing

**来源**：`feedback_quality_gate`

---

### F9：CLI 单元测试要覆盖边界

**规则**：任何 CLI 命令，至少 1 个 happy path + 1 个 error case + 1 个边界条件。

**Why**：CLI bug 用 skill 能识别，单元测试覆盖边界能提前发现。

**How to apply**：
- Happy path：正常输入 → 正常输出
- Error case：非法输入 → 错误信息
- 边界：空字符串 / 超长字符串 / 特殊字符

**来源**：`feedback_cli_bug`

---

### F10：描述流程只引文档，不按源码拼凑

**规则**：描述流程只能引权威文档（如 SKILL.md L41），不要按源码拼凑。

**Why**：第 5 次被骂。按源码拼凑容易出错（拼错行号/拼错顺序），引文档更可靠。

**How to apply**：
- 写流程描述时，找权威文档原文引用
- 文档更新时同步更新流程描述
- 不确定时优先查文档，不查源码

**来源**：`feedback_prismos_canonical_flow`

---

### F11：用户边界 = 由用户定义

**规则**：不要替用户定义"想写什么"；不要用"项目边界"逻辑限制用户的选题方向。

**Why**：用户用工具是为了"想写什么就写什么"，工具边界 = 用户定义。

**How to apply**：
- 用户给命题 → 工具接住，不要挡回去
- 工具的能力上限是"做不到"而非"不该做"
- 例外：违法/违规/伤害他人的内容，工具应该拒绝

**来源**：CLAUDE.md 错误 3

## 2. CLAUDE.md 规定的开发习惯（D1-D9）

### D1：编码前先思考

不臆断、不隐藏困惑、明确权衡。在实现之前：
- 明确说明你的假设。如果不确定，请询问。
- 如果存在多种解释，请一一列出。
- 如果存在更简单的方案，请说明。
- 如果有不清楚的地方，请停止。指出令人困惑之处。
- 回顾历史教训：上一次的同类错误是什么？这次如何避免重蹈覆辙？

---

### D2：简单至上

用最少的代码解决问题，不进行推测性编码。
- 不提供超出要求的特性
- 不为仅使用一次的代码进行抽象
- 不提供未经要求的"灵活性"
- 不为不可能发生的场景编写错误处理
- 200 行能精简到 50 行就重写

**代码质量原则**：
- 组件解耦：每个功能模块独立
- 单向依赖：避免循环依赖
- 单一职责：每个组件/函数只负责一件事
- 可独立测试
- 避免过度设计

---

### D3：精准修改

只触碰必须修改的部分，只清理你自己留下的烂摊子。
- 不"改进"相邻的代码或格式
- 不重构没有问题的代码
- 遵循现有风格，即使你会有不同的做法
- 发现死代码请指出，不直接删除
- **不重复犯错**：同一个错误不犯第二次

---

### D4：目标驱动执行

定义成功标准，循环执行直到验证通过。
- "添加校验" → "编写测试，然后使其通过"
- "修复 Bug" → "在测试中复现它，然后修复"
- "重构 X" → "确保重构前后的测试都能通过"

---

### D5：TDD（测试驱动开发）

1. 写任何代码之前，**先写失败的单元测试**定义预期行为
2. 实现最小代码让测试通过
3. 每次 Edit/Write 操作后运行测试
4. 测试失败时自动修复并重新测试（最多 3 次）
5. 报告最终测试结果和具体失败项

---

### D6：并行代码库探索

在做出任何代码更改之前，**并行**运行：
1. Glob 查找所有相关文件类型
2. Grep 查找导入/导出语句和函数签名
3. Grep 查找现有类似实现
4. 读取关键配置文件

综合以上结果，再动手。

---

### D7：Git 规范

- 直推 main：小功能直接推到主分支，不走 PR 流程
- Commit 格式：`feat: xxx` / `fix: xxx` / `refactor: xxx`
- Commit 前检查：代码能跑、测试通过

---

### D8：尽量少打扰用户

- 开发过程中自动推进，不问"要不要继续"
- 只在关键节点问：TDD 选择 / 技术栈多选项 / 真正卡住

---

### D9：持续更新 MEMORY

- 每个阶段完成后写入断点记录
- 项目 MEMORY 放 `~/.claude/projects/<project>/memory/projects_<name>.md`
- Git history 是历史记录，不单独写 changelog

## 3. 炉项目专属决策（P1-P14）

### P1：物理隔离

源项目 + 新项目独立迭代。不同 git 仓库，代码复制一次性，不双向同步。
- 日期：2026-06-15
- 原因：用户明确"原先两个项目仓库的代码不要动"

---

### P2：分阶段推进

- v1：单用户（本地 YAML 配置）
- v2+：单用户延伸（飞书表格管理配置 + 跨设备同步）
- v3 长期：开放给其他人使用（账号/权限/多租户/Web 化/计费）

---

### P3：核心能力必须继承

炉要覆盖源项目核心能力 + 融合 + 衍生新方向：
- 选题（PRISM 苏格拉底追问 + CCOS + Gap）
- 原创（ContentForge create）
- 二创（ContentForge recreate + PRISM ViralGenome）

---

### P4：8 维度评分每项 ≥ 7.5

- 6 维：温度/热度/深度/厚度/情绪曲线/知识迁移（继承 ContentForge）
- L5 三项：观点锐度/思想模型应用/事实准确性（炉新增）
- 验收 = 8 维度每项 ≥ 7.5 + 用户亲自读完判断

---

### P5：7 步循环

命题输入 → 追问澄清 → 蓝图设计 → 段位选择 → 草稿生成 → 打磨质检 → 沉淀回写

- 灵感拆 2、框架拆 2，其他不动
- 7 步每步单一职责，错误易定位
- 用户参与 ≤ 1h；系统时间要快

---

### P6：5 种策略模式

思想模型框架的策略模式：
- chain（A→B→C 链式）
- parallel（A‖B‖C 并行）
- nested（A(B(C)) 嵌套）
- divergent→convergent（发散→收敛）
- condition（条件分支）

v1 跑 5 种，iterate/recursive 留 v2+。

---

### P7：不硬编码，可配置

- 思想模型/框架/必避免列表走 YAML（v1）
- 飞书多维表格（v2+）
- 配置路径不绑死，可本地可云端

---

### P8：可复用为目标

文档可项目内 + 其他项目参考：
- 99-LESSONS-LEARNED.md 通用化
- 架构/模块设计有"为什么这样设计"的说明
- ADR 记录关键决策

---

### P9：避免重跑错方向

6 层面（边界/目标/架构/模块/技术/文档）先对齐再开发，避免上来就实施。
- 边界：什么/不是什么
- 目标：核心痛点/验收标准/时间
- 架构：5 步循环/8 段/Anti-AI
- 模块：苏格拉底追问/思想模型/Blueprint
- 技术：Python/lark-cli/持久化
- 文档：PRD/规范/计划/测试

---

### P10：4 思想模型框架

| 框架 | 策略 | 模型 |
|------|------|------|
| 问题解构 | chain | 第一性原理 + 5Why + 奥卡姆剃刀 |
| 决策分析 | parallel | 杠杆者 + JTBD + 反共识 |
| 系统思考 | nested | 系统思维 + 因果链 + 反馈回路 + 二阶效应 |
| 创新突破 | divergent→convergent | 逆向 + 类比 + 第一性原理 + 反共识 |

框架 = 4 要素：Trigger / Strategy / Models with role / Output contract

---

### P11：8 项追问产出

苏格拉底追问的输出格式：
1. 命题浅层
2. 底层逻辑
3. 潜在诉求
4. 风格建议
5. 反共识点候选（2-3 个）
6. 思想模型框架候选（1 主 1 备）
7. 风险点/边界
8. 可证伪性

---

### P12：L5 三子项

| 子项 | 检测项 |
|------|--------|
| 观点锐度 | 强观点密度 / 模糊词密度 / 关键判断数 |
| 思想模型应用 | 模型调用识别 / 贴合度 / 解释度 |
| 事实准确性 | 可追溯 / 虚构警告 / 引用准确 |

每子项独立 0-10 分。纯举例/纯客观可虚构，核心事实必追溯。

---

### P13：必避免列表

风格画像的字段之一：
- AI 套话列表（"赋能""在这个时代"）
- 用户禁忌（动态更新）
- 通过 L1 禁词扫描 + L2 风格检查执行

---

### P14：苏格拉底追问 3 阶段学习

学习"什么情况算够了"：
- 阶段 1（0-30 样本）：靠用户说"够了"停 + 记录上下文
- 阶段 2（30-100 样本）：系统提示"可收尾" + 用户决定
- 阶段 3（100+ 样本）：自动判断，不再问

存储到 `style_profile.socratic_stop_signal`。

## 4. 通用经验（可在其他项目复用，E1-E5）

### E1：文档先于代码

PRD / 架构 / 模块设计 / 测试方案 优先于实施，文档不全不开工。

### E2：决策记录（ADR）

关键设计决策写 ADR，格式：背景 / 备选 / 选定 / 后果 / 日期。

### E3：进度透明

TASKS / PROGRESS / MILESTONES 持续更新，任何时候接手都能恢复上下文。

### E4：问题清单

ISSUES 文件持续记录：优先级 / 状态 / 解决方案。

### E5：技术债务清单

已知 hack / 待重构单独记录，避免技术债务累积到不可控。

## 5. 维护规范

### 新增教训的流程
1. 用户纠正 / 验证后，立刻记一条
2. 格式：规则 + Why + How to apply + 来源
3. 编号续上（F12, D10, P15, E6...）

### 复盘频率
- 每 2 周：看是否有重复出现的错误
- 每里程碑：整合同类教训为"原则"

### 跨项目复用
- 任何项目开始前，先读本文件
- 99 编号保留，新项目不重置编号
- 项目专属教训单独写 `projects_<name>.md`，不污染通用清单
