# D-001: 独立仓库策略（物理隔离）

## 状态
已敲定 | 2026-06-15

## 背景
炉 是新项目，从 ContentForge（TypeScript）+ PRISM-OS（Python）两个源项目融合而来。需要决定：
- 源项目 + 炉 的关系
- 代码归属与维护
- 未来能力演进

## 备选方案

### 方案 A：替代
- 源项目代码迁到 炉
- 源项目废弃
- 炉 接管所有能力

### 方案 B：并存（独立迭代）
- 源项目 + 炉 各自独立仓库
- 各自继续迭代
- 能力借鉴 = 手动复制

### 方案 C：吸收
- 源项目代码迁到 炉
- 源项目代码作为"参考"存在
- 新能力都进 炉

## 选定
**方案 B：并存（独立迭代）**。

理由：
1. 源项目有独立用户群，废弃会破坏它们
2. 物理隔离避免误改源项目
3. 能力借鉴通过手动复制，可控可追溯
4. 用户明确"原先两个项目仓库的代码不要动"

## 后果

### 正面
- 源项目继续维护，互不影响
- 炉 独立演进，无历史包袱
- 代码库边界清晰

### 负面
- 需要手动同步能力（费时）
- 两个项目文档要分别维护
- 升级源项目时需手动复制到 炉

### 风险
- 源项目新能力不会自动到 炉
- 缓解：定期看源项目 release notes

## 实施

- 源项目代码快照到 `docs/legacy/contentforge/` + `docs/legacy/prismos/`
- 不带 `.git` 目录（一次性快照）
- 后续不双向同步
