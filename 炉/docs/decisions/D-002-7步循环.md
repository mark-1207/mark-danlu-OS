# D-002: 7 步循环

## 状态
已敲定 | 2026-06-15

## 背景
创作流程要拆成几步，每步单一职责、可定位、可中断。需要决定：
- 步数（粗 vs 细）
- 每步边界
- 用户参与点

## 备选方案

| 方案 | 步数 | 评估 |
|------|------|------|
| A | 5 步（灵感/框架/草稿/打磨/沉淀） | "灵感"和"框架"承载过重 |
| B | 6 步（+ 段位选择） | 草稿/打磨粒度仍粗 |
| **C** | **7 步（命题输入/追问/蓝图/段位/草稿/打磨/沉淀）** | **每步单一职责** |
| D | 8 步（+ 沉淀细分） | 用户中断次数过多 |

## 选定
**方案 C：7 步**。

理由：
1. 灵感拆 2（命题输入 + 追问）— 命题标准化和认知澄清分清楚
2. 框架拆 2（蓝图 + 段位）— 大纲和段位分开
3. 草稿/打磨/沉淀不动
4. 错误易定位
5. 用户参与 ≤ 1h（CLAUDE.md 错误 0 原则）

## 后果

### 正面
- 每步 LLM 任务聚焦
- 错误易定位
- 用户中断次数适中（7 次）

### 负面
- 步数比 5 步多（用户操作 +2）
- 实现复杂度高

### 风险
- 7 步都可能中断，需要 persist 机制
- 缓解：[02-ARCHITECTURE 第 4 节](02-ARCHITECTURE.md) 已设计

## 实施

```
命题输入 → 追问澄清 → 蓝图设计 → 段位选择 → 草稿生成 → 打磨质检 → 沉淀回写
```
