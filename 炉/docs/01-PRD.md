# 01 — 产品需求文档（PRD）

> 版本：v1.0 草稿 | 状态：方案已对齐，文档完整

## 1. 产品定位

### 1.1 一句话
**炉** = 你的内容风格化创作引擎。从一个命题到一篇有"你"的声音的文章，7 步闭环跑通。

### 1.2 核心问题（要解决什么）

| 问题 | 表现 | 优先级 |
|------|------|--------|
| 有了选题但写不出来 | 命题有了，落笔就卡 | P0 |
| AI 味儿太浓 | 套话/节奏呆板/无个人视角/比喻陈旧/无反共识/数据堆砌 | P0 |
| 个人风格难沉淀 | 这次写得好，下次又变 AI 味儿 | P0 |
| 质量不稳定 | 时好时坏，依赖灵感和状态 | P1 |
| 认知不清 | 命题模糊就开始写，写完才发现在论证"错的东西" | P1 |

### 1.3 产品愿景
- **短期**：让 mark 写文章的质量稳定在 8+ 维度评分
- **中期**：让方法论可复用（不同命题类型可批量跑）
- **长期**：让"个人风格"成为可沉淀的资产（v3 长期，开放后多人复用）

## 3. 核心能力（详细）

### 3.1 苏格拉底追问
- **目的**：认知澄清，确保命题真正想清楚
- **输入**：用户原始命题
- **输出**：8 项产出（详见 P11）
- **机制**：3-5 轮固定模板 + 动态调整；学习"什么情况算够了"（3 阶段）

### 3.2 思想模型框架
- **目的**：用结构化思维工具拆解命题
- **输入**：命题 + 追问产出
- **输出**：选定的框架 + 模型组合
- **机制**：4 内置框架（问题解构/决策分析/系统思考/创新突破）+ 12 模型卡片
- **5 种策略模式**：chain / parallel / nested / divergent→convergent / condition

### 3.3 7 步循环
```
命题输入 → 追问澄清 → 蓝图设计 → 段位选择 → 草稿生成 → 打磨质检 → 沉淀回写
```

- **目的**：把"创作"拆成可执行、可验证、可干预的 7 步
- **每步单一职责**：错误易定位；用户操作 ≤ 1h
- **可中断/可续跑**：任何一步中断后可续跑（见 3.10 状态机）

### 3.4 8 段弹性结构
- **核心 5 必选**：钩子 / 反共识 / 案例 / 思想模型 / 金句收尾
- **可选 8 项**：行动指南 / 反驳 / 对比 / 数据 / 自嘲 / 引用 / 转折 / 留白
- **按内容类型推荐 2-4 段**：用户确认调整
- **短/中/长**：
  - 短文（≤500字）：核心 5 选 3
  - 中篇（500-2000字）：核心 5 + 可选 1-2（共 6-7 段）
  - 长篇（>2000字）：核心 5 + 可选 2-4（共 7-9 段）

### 3.5 L5 三子项评分

| 子项 | 检测 |
|------|------|
| **观点锐度** | 强观点密度 / 模糊词密度 / 关键判断数 |
| **思想模型应用** | 模型调用识别 / 贴合度 / 解释度 |
| **事实准确性** | 可追溯 / 虚构警告 / 引用准确 |

每子项独立 0-10 分。配合原有 6 维（温度/热度/深度/厚度/情绪曲线/知识迁移）= 8 维度评分。

### 3.6 飞书工作流
- **灵感池**：原始洞察录入
- **反馈数据**：实际表现回写（手动录入）
- **v2+**：飞书多维表格管理配置

### 3.7 Obsidian 素材库
- **本地知识库**：跨命题复用的案例/数据/金句/思想模型
- **结构**：输入层 / 提炼层 / 创作层 / 输出层
- **沉淀路径**：草稿完成 → 用户标记"采纳" → 写入 Obsidian

### 3.8 爆款二创
- **输入**：已有爆款文章（文件或 URL）
- **流程**：ViralGenome 拆解 → 差异化重写 → 草稿 → 质检
- **v1 简化版**：3 步（拆解 / 重写 / 质检），不循环迭代

### 3.9 风格画像
- **存储**：本地 YAML（v1）/ 飞书表格（v2+）
- **字段**：
  - 语言指纹（比喻偏好/节奏/口头禅）
  - 必避免列表（AI 套话 + 用户禁忌）
  - 追问停顿时机信号（阶段 1-3）
  - 风格偏好历史

### 3.10 状态机
- **每步可中断**：context.persist() 落盘
- **续跑机制**：`--resume <run_id> --from-step <step>`
- **失败重试**：单步失败可重试，跨步状态保留

## 4. 典型用例

### 4.1 场景 A：从命题到成文

```
1. mark 给出命题："AI 牛马陷阱"
2. 炉 苏格拉底追问 3 轮 → 8 项产出
3. 炉 选框架（决策分析）→ 蓝图设计
4. mark 确认段位选择
5. 炉 草稿生成（7-8 段）
6. 炉 打磨质检（8 维度评分）
7. mark 读完 + 修改 + 标"采纳"
8. 炉 沉淀（写回风格画像 + Obsidian）
```

### 4.2 场景 B：从爆款到二创

```
1. mark 准备爆款文章文件
2. 炉 ViralGenome 拆解
3. 炉 差异化（结合 mark 风格画像 + 思想模型）
4. 炉 重写草稿
5. 炉 质检 + mark 验收
6. 沉淀
```

### 4.3 场景 C：风格画像迭代

```
1. mark 在风格画像中加新禁忌
2. 下次写时必避免列表自动更新
3. mark 跑几篇后，AI 味儿命中率下降
4. mark 写一段金句采纳 → 写入 Obsidian 金句库
5. 下次写时可用作参考
```

## 5. 验收标准

### 5.1 硬指标
- **8 维度评分**（6 维 + L5 三项）每项 ≥ **7.5**
- 草稿生成一次通过率 ≥ 80%（v1 目标）

### 5.2 软指标（最高标准）
- **mark 亲自读完**判断"这就是我的声音" — **人工验收为最高标准**
- 必避免列表命中次数 ≤ 2/篇

### 5.3 时间
- 涉及人的参与（修改内容等）≤ 1 小时
- 系统时间要快：
  - 7 步全跑 ≤ 10 分钟（不含人工）
  - 单步平均 ≤ 90 秒

### 5.4 验收流程
1. 炉 跑完 7 步 + 输出 8 维度评分
2. mark 亲自读完 + 标注
3. 如未达 7.5：炉 自动给出"修复建议"+ mark 决定重跑/接受/手动改
4. 达 7.5：进入"沉淀"

## 8. 风险与缓解

### 8.1 技术风险

| 风险 | 等级 | 缓解 |
|------|------|------|
| LLM 不可控导致质量波动 | 高 | 8 维度评分 + 必避免列表 + 多层 prompt 校验 |
| LLM 调用成本失控 | 中 | 7 步 ≤ 15 次 LLM 调用（v1 控制） |
| 思想模型库质量不足 | 中 | 内置 12 模型卡片，每卡有正反例 + 常见错用 |
| 持久化失败导致数据丢失 | 中 | 每步 persist + 版本快照 |

### 8.2 产品风险

| 风险 | 等级 | 缓解 |
|------|------|------|
| 用户用着用着不用了 | 中 | 必避免列表 + 风格画像积累 → 沉没成本 |
| 自动化过度，失去"我"的声音 | 中 | mark 亲自验收为最高标准 |
| 8 维度评分标准僵化 | 低 | L5 子项可调，6 维可调 |

### 8.3 用户风险
- v1 单用户风险低
- v3 多用户时：账号安全 / 数据隔离 / 滥用

### 8.4 运营风险
- LLM Provider 宕机 → 链路 fallback（v2 评估）
- 飞书 API 限流 → 本地缓存

- v1 暂不接入（先手动管理素材）

- **Python 版本**：3.11+

## 10. 度量与监控

### 10.1 核心指标（产品）
- **质量**：8 维度评分平均
- **效率**：7 步全跑时间
- **使用率**：每周使用次数
- **沉淀**：Obsidian 素材库每周新增

### 10.2 健康指标（系统）
- **LLM 调用次数/篇**：≤ 15
- **必避免列表命中率**：≤ 2 次/篇
- **单步失败率**：≤ 5%

### 10.3 报警阈值
- LLM 调用成本 > 单篇预算 → 警告
- 8 维度评分 < 7.5 → 必须人工验收
- 必避免列表命中率 > 5/篇 → 标记"风格画像漂移"
