# Claude Code 项目指南

## 一般指南

### 写代码前确认方案
在编写任何代码之前，向用户明确确认你的方法。当不确定他们偏好什么时，提出2-3个具体选项，而不是随意选择。

示例：
- "我可以用A、B或C三种方式实现这个功能。你更倾向哪种？A方案...，B方案...，C方案..."

## 代码质量

### 最小粒度开发（最高优先级）
**任何功能开发必须遵守此规则**：

1. **组件解耦**：每个功能模块应该是独立的组件，修改一个组件不应该影响其他组件
2. **单向依赖**：组件依赖关系应该是单向的，避免循环依赖
3. **单一职责**：每个组件/函数只负责一件事
4. **可独立测试**：每个组件应该可以独立运行和测试
5. **避免过度设计**：不要创建"可能有用"的抽象，只创建当前需要的

**具体做法**：
- 新功能先创建独立组件文件，不要直接写在主页面中
- 组件间通信使用props/回调，不使用共享状态
- 如果多个页面需要相同功能，创建可复用的独立组件
- 修改现有组件时，先确认修改范围会不会影响其他功能

**违反规则的后果**：代码耦合导致无法修改、回滚困难

### 增量测试UI组件
实现UI组件时，逐步测试——做一次更改，验证它有效，然后再继续。避免进行可能引入新bug的多个相互依赖的更改。

对于UI任务，告诉用户："我先做这个修改，你可以验证它有效后我再继续下一个调整。"

## 沟通

### 始终提供文件路径
当命令输出文件路径时，在响应中提供确切的文件路径。不要让用户运行命令来查找它。

## 常用工作流

### 测试驱动开发
1. 在写任何代码之前，先写失败的单元测试来定义预期行为
2. 实现最小代码通过测试
3. 每次Edit/Write操作后运行测试
4. 如果测试失败，自动修复并重新测试（最多3次）
5. 报告最终测试结果和具体失败项

### 并行代码库探索
在做出任何代码更改之前，并行运行探索：
1. Glob查找所有相关文件类型（.ts, .py, .json）
2. Grep查找导入/导出语句和函数签名
3. Grep查找现有类似实现
4. 读取关键配置文件

使用多个Bash/Grep命令同时进行并行操作。在建议任何更改之前，报告你发现的摘要。

### Windows环境预检
在进行任何文件删除、文件夹重命名或技能安装之前：
1. 检查目标目录是否在终端中当前打开（pwd匹配）
2. 检查运行中的开发服务器（netstat检查常见端口3000,5173,8000）
3. 检查文件锁
4. 对于技能，验证Windows兼容性和所需依赖

报告发现的任何阻止项并建议修复方法（关闭终端、终止进程、安装依赖），然后再继续。
